# 测试快照 v2026.07.24：当前系统如何工作

> 本文对应当前云端测试快照，基于 `cloud-server-v1.1.24` 的代码继续开发。它描述的是**测试流程和测试数据展示**，不是对外医疗产品说明，也不构成诊断、治疗或处方建议。

## 1. 这版要解决什么问题

这版把“有一堆算法数值”改成一条可追溯、可解释的测试链路：

1. 读取设备上传的 EEG、EMG、IMU 数据包；
2. 使用 GPU 上的评分模型得到 FMA 手部子量表、手部肌张力和 Brunnstrom 手功能分期模型预测；
3. 计算 26 项脑电、肌电、脑肌和运动学指标；
4. 用 `planner_rag` 组织事实、知识检索和报告文本；
5. 把每项指标展示成“本次结果｜解读｜依据”，而不是把原始数值、文献边界和长段说明混在一个格子里；
6. 以 Brunnstrom 手功能模型预测为核心，生成一个明确标记为“测试性归纳”的综合亚型；
7. 在知识库测试页按“指南/共识优先、教材作背景、论文作补充”的顺序展示证据来源。

当前全部内容都可用于系统联调、界面验证和证据展示；专家审核是后续治理工作，不阻塞当前测试功能。

## 2. 运行结构

```text
浏览器
  │
  ▼
Nginx :6006
  ├─ frontend/dist（React/Vite 构建产物）
  └─ /api → FastAPI :8000
                 │
                 ├─ MySQL：测试档案、评估记录、原始报告和导出元数据
                 ├─ GPU 评分模型：FMA / Hand MAS / Brunnstrom
                 ├─ planner_rag：解释、检索计划、证据检索、报告生成与校验
                 └─ RAG 测试服务 :8011：知识库检索与来源详情
```

- 评分和报告模型必须由带 CUDA 的 Python 环境加载；代码会在没有 GPU 时明确报错，避免悄悄退回 CPU 推理。
- 前端静态文件由 Nginx 直接提供；后端代码更新后使用标准后端重启脚本加载。
- MySQL 保存测试记录；前端对旧测试文本采用兼容显示，不修改旧报告原文。

## 3. 从信号到报告的测试流程

### 3.1 输入与评分

设备上传的数据包经解析和质量检查后，进入同一条异步队列。评分模块输出：

- FMA 手部子量表模型预测（0–20 分）；
- Hand MAS 手部肌张力模型预测；
- Brunnstrom 手功能分期模型预测；
- 26 项 EEG、EMG、脑肌和 IMU 指标。

量表结果在页面中始终标为“模型预测值”，不是人工现场评定。

### 3.2 `planner_rag` 编排

报告不直接把模型输出交给大模型自由生成，而是按固定顺序运行：

```text
质量门控
  → Interpreter（把每个数值变为结构化 finding）
  → CoreKnowledgeProvider（按 system_key 绑定核心知识）
  → KnowledgePlanner（只规划检索主题，不下结论）
  → Retriever（一次批量检索）
  → ReportGenerator（只生成摘要、限制和康复方向）
  → Validator（引用、模型预测标识、确定性诊断、药物和精确剂量检查）
  → Markdown 渲染
```

其中数值、观察项与来源编号由确定性代码装配；大模型不负责复制全部指标表，也不能凭空制造来源编号。这样可以把“模型生成的文字”和“本次实际输入、证据来源”分开追溯。

### 3.3 证据与引用

- 每个 `finding` 会关联固定核心知识和/或本次检索结果中的来源 ID。
- 报告的来源编号由代码去重并统一编号，前端和导出文件使用同一组编号。
- 检索不可用时，仍可使用已绑定的核心知识解释边界；不会伪造外部检索结果。
- 验证器会检查未知来源、遗漏的输入 finding、模型预测未标注、确定性诊断、药物建议和精确训练剂量等问题。

## 4. 指标表：现在如何给人看

新测试结果统一使用四列：

| 指标 | 本次结果 | 解读 | 依据 |
|---|---|---|---|
| 运动平滑度（SPARC） | 本次记录值 | 用于观察动作是否连续、流畅，是否频繁停顿或抖动；该值是本设备/算法计算的本次记录，暂无统一好坏范围，后续在同设备、同动作条件下复测看变化。 | 来源编号 |

### 4.1 本次结果

只放事实：`模型预测值` 或 `本次记录值`，包含单位，不混入文献解释。

### 4.2 解读

按固定顺序输出：

1. **这个指标检测什么**；
2. **本次数值是什么性质**（模型预测、设备计算值、文献范围内/高于/低于等）；
3. **如何理解本次值**（例如作为个人基线、看同条件复测变化，或结合动作表现）。

系统已为 26 项指标补充用途映射。例如：

- 皮层-肌肉相干：看大脑运动区和肌肉发力是否同步配合；
- 腕/指共收缩：看屈伸肌是否同时用力、动作是否协调；
- 积分肌电：看相应肌肉在整个动作中的总用力量；
- 中位频率：看相应肌肉的疲劳或募集变化；
- 运动平滑度：看动作是否连续、流畅；
- 关节活动度代理：看本次动作活动范围的大小；
- 峰值角速度：看对应腕屈、腕伸或伸指动作达到的最快速度。

对于没有统一绝对阈值的设备/协议特异指标，系统不会把一次结果标成“正常”或“异常”；但会先说明它测什么，再说明本次数值是个人基线，避免只留下空泛的免责声明。

### 4.3 历史测试结果兼容

历史 `planner_rag` 文本原来是三列“指标｜本次结果与知识解读｜依据”，而且没有综合亚型。前端在渲染时：

- 从旧表格中拆出本次结果和短解读；
- 从 Brunnstrom 模型预测行读取分期，补出“综合亚型（历史测试结果补全）”；
- 不改写 MySQL 中保存的原始 Markdown。

因此刷新页面即可看到新布局，无需重跑旧测试任务。

## 5. 综合亚型如何生成

新生成的 `planner_rag` 测试报告在“综合评估结果”后加入“综合亚型界定”。

- 以 Brunnstrom 手功能模型预测分期为主线；
- 结合 FMA 手部子量表模型预测；
- 写成“手功能综合亚型（测试性归纳）”；
- 明确说明中枢驱动、协同分离和关节活动度仍需动作检查确认。

这不是确定性医学诊断。它的作用是把分散的测试结果整理成一个便于页面浏览的总括标签。

## 6. 知识库测试页如何工作

### 6.1 证据层级

知识库测试优先展示：

1. 公开临床指南、专业共识和官方标准；
2. 权威教材的书目与章节定位，作为背景知识；
3. 论文作为补充证据。

当前测试索引包含 RCP 卒中指南、澳大利亚卒中临床标准、澳新 Living Guideline 等公开官方来源；Braddom 和 DeLisa 教材只保存书目信息、章节定位和获取方式，不复制受版权保护全文。

### 6.2 来源卡片

每条检索命中返回独立来源详情：

- 来源类型、证据层级与权重；
- 原始 HTTPS 链接；
- 访问状态与版权边界；
- 系统内短证据摘录。

系统目前保存的是短摘录和可追溯链接，**不是**受版权保护资料的完整本地副本。公开或已授权资料如需做本地原文归档，应另行建立归档清单、哈希和受鉴权访问路径。

### 6.3 测试边界

检索页明确标记为 `test_only`；未审核证据可以在测试页展示，但保留来源、证据层级、适用范围与版权说明。专家审核计划保留为后续工作，不作为当前测试页面的前置条件。
